\begin{abstract}
This guide tells a team how to \emph{build} \amos. It converts the research paper's memory-operating kernel into a product architecture, a ruthlessly bounded first production slice, normative contracts for that slice, and exit gates that prove the slice works before enterprise surface area is added.

\amos sits between analytical agents and governed data systems. It compiles a permission-safe working set, verifies computation against current schema and metric state, executes through capability-limited workers, and commits every material claim with provenance, review state, invalidation rules, and replay metadata. Models propose plans and interpretations; deterministic services retain authority over access, execution, validation, and publication.

The architecture is deliberately incremental. A startup first ships a modular monolith that completes one read-only analytical workflow end to end. It then adds durable feedback, scheduling, multi-tenant governance, connector breadth, and controlled autonomy only after explicit correctness, security, reliability, and customer-value gates pass. The guide defines the complete subsystem map and the normative contracts that detailed designs must implement. Appendices A--F freeze the most dangerous first-release contracts. Appendices G--K describe the current Rust implementation: its complete type and function inventory, exact persistence and transport surfaces, test evidence, low-level cost model, and known production gaps. Database DDL, generated schemas, and adapter conformance fixtures remain versioned engineering artifacts rather than claims hidden in prose.
\end{abstract}

\section{Product Value and System Boundary}
\label{sec:value}

\subsection{The job to be done}

An analyst or scheduled review must answer a recurring operational question---for the first slice, payment-failure health---faster than the status quo \emph{and} with inspectable state validity. The product succeeds when a reviewer can open one artifact and answer: what changed, which metric and schema governed the numbers, whether permission and freshness held, what remains review-gated, and what a correction will affect next.

\must The first commercial workflow is a payment-health or subscription-health review: recurring, economically meaningful, bounded, and reviewable. A fluent report without admitted transaction, context manifest, verified executions, structured claims, dependencies, replay package, audit trail, and explicit terminal outcome is not an \amos product slice.

\subsection{What AMOS is}

\amos is the control and memory plane for autonomous data analysis. It is not a warehouse, dashboard, general document search engine, or particular language model. It converts changing schemas, metrics, snapshots, documents, permissions, prior analyses, reviewer corrections, and execution records into governed analytical memory, then runs one stable protocol over that memory.

Three surfaces share one substrate:
\begin{enumerate}
  \item \textbf{Analytical memory:} typed, versioned objects with authority, effective time, permissions, supersession, and provenance.
  \item \textbf{Agent runtime:} governed planning, read-only SQL, deterministic statistics and charts, verification, artifact construction, review, and replay.
  \item \textbf{Application platform:} APIs and templates for the first review workflow; later monitors, forecasts, and domain agents reuse the same substrate.
\end{enumerate}

\subsection{What AMOS is not}

\nongoal Replacing warehouses, semantic layers, catalogs, lineage systems, or BI tools. \nongoal Unrestricted natural-language access to all company data. \nongoal Arbitrary production writes. \nongoal Perfect model reproducibility. \nongoal Automatic causal discovery. \nongoal Universal enterprise knowledge management.

Source systems own raw data, primary access control, warehouse transactions, catalog records, semantic definitions, document originals, and identity truth. Applications own user experience and domain workflow. Models own no durable authority: model output is a proposal until a deterministic gate accepts it.

\subsection{Why an operating layer is required}

At time $t$ an analytical task depends on independently changing state:
\begin{equation}
  \mathcal{T}_t = (q_t, D_t, C_t, M_t, X_t, H_{<t}, F_{<t}, P_t),
\end{equation}
where $q_t$ is the request, $D_t$ data snapshot or stream position, $C_t$ schema/catalog state, $M_t$ metric/semantic state, $X_t$ governed documents, $H_{<t}$ prior work, $F_{<t}$ reviewed feedback, and $P_t$ identity and policy.

These sources do not share one transaction manager. A metric can change after retrieval, a table can migrate during execution, a watermark can advance, and a permission can be revoked before publication. Prompt context, RAG, and text-to-SQL address fragments of the problem; they do not provide a durable operating abstraction for analytical state. That is the research paper's core claim and this guide's build target.

\section{Kernel Contracts from the Research Paper}
\label{sec:kernel}

This section freezes only the research contracts a builder must implement. Formal motivation and evaluation live in the research paper.

\subsection{Memory object}

Persistent analytical memory is a set of typed objects. Each object carries stable identity, type, structured content or external reference, summary, source, authority, effective interval, recording time, permissions, sensitivity, version, status, supersession links, provenance reference, and content hash. Logical identity and physical version are separate: multiple versions may describe one metric or schema, but only versions valid for the task interval and policy may govern a run.

Core types for the first slice: data/stream state, schema, semantic definition, governed document, prior analysis, feedback, permission policy, execution, artifact, claim, and provenance edge. Raw fact tables remain external.

Default authority order:
\[
\text{owner-approved} > \text{reviewer-approved} > \text{user note} > \text{system-observed} > \text{model hypothesis} > \text{untrusted external}.
\]
Effective time determines when a statement applies; recording time determines when \amos learned it. Equal-authority conflicts for the same interval block publication; they are never silently resolved by recency.

\subsection{Publication invariants}

Let $W_t$ be the model-visible working set, $E_t$ the executable computation, $Y_t$ the artifact, and $P_t$ the active policy:
\begin{align}
I_1 &: \forall i \in W_t,\; permitted(i,P_t), \\
I_2 &: \forall i \in W_t,\; effective(i,t) \land \neg superseded(i,t), \\
I_3 &: execute(E_t) \Rightarrow schemaValid(E_t,W_t) \land metricValid(E_t,W_t) \land readOnly(E_t), \\
I_4 &: publish(Y_t) \Rightarrow \forall c \in importantClaims(Y_t),\; supported(c,\mathcal{G}_t), \\
I_5 &: replayable(Y_t) \Rightarrow complete(R_t,W_t,E_t,\mathcal{G}_t).
\end{align}
$I_1$ is enforced before context construction; $I_2$ during retrieval and reconciliation; $I_3$ before tool execution; $I_4$--$I_5$ before publication.

\subsection{Primitives and outcomes}

Stable primitives: \code{retrieve}, \code{write}, \code{supersede}, \code{reconcile}, \code{compact}, \code{verify}, \code{cite}, \code{replay}. Storage, retrieval backends, and model providers are replaceable behind these primitives.

Every task ends in an explicit outcome. Silence and best-effort finalization are forbidden.

\centering

\begin{tabularx}{\linewidth}{L{0.17\linewidth}L{0.24\linewidth}Y}
\toprule
\textbf{Outcome} & \textbf{Meaning} & \textbf{Required behavior} \\
\midrule

\subsection{Consistency classes}

Connectors advertise observation strength only after certification:
\begin{itemize}
  \item \textbf{C0 observed:} version and observation time recorded; re-readability not guaranteed.
  \item \textbf{C1 validated:} version can be re-read or compared at commit.
  \item \textbf{C2 retained:} exact version retained or leased for deterministic reconstruction.
\end{itemize}
\must A timestamp label alone is never a version guarantee. The first slice requires at least C1 for metric, schema, and policy roles, and C2 for the warehouse snapshot used by replay-level-3 artifacts.

\section{First Production Slice}
\label{sec:slice}

\subsection{In scope}

\must The first production slice supports:
\begin{itemize}
  \item one tenant;
  \item one read-only warehouse connector;
  \item one approved metric family (\code{payment\_failure\_rate});
  \item three query shapes (summary rate, concentration, time series);
  \item deterministic built-in statistics and one structured chart;
  \item governed documents and reviewer feedback;
  \item typed report claims constructed before prose render;
  \item reviewer approve / reject / correct;
  \item reverse-dependency invalidation for metric, schema, permission, and data-state changes;
  \item computational replay with hash comparison (level 3 for the payment artifact).
\end{itemize}

The synthetic payment scenario from the research paper is the reference fixture: twelve hours of payment events, a six-hour failure spike, Processor~B / Visa concentration, excluded test accounts, a deployment note, watermark lag, schema migration from \code{failure\_reason} to \code{error\_code}, restricted prior incident, and approved metric v3.

\subsection{Cut until earned}

\cut General Python workers, free-form claim extraction from prose, vector indexes, autonomous external publication, multi-tenant SSO fleets, customer-managed keys, regional data planes, dedicated worker pools, hybrid retrieval, portfolio scheduling, and controlled autonomy. These appear only after the gates in Section~\ref{sec:build}.

\deferred from the first slice but specified lightly for continuity: PostgreSQL/RLS production adapter, object-storage blob backend, secret manager, and real warehouse credentials. Domain ports must not depend on those adapters.

\subsection{Definition of a successful run}

A request or schedule is complete only when the system persists all of the following:
\begin{enumerate}
  \item admitted \atxn with identity, risk, budgets, and policy epoch;
  \item connector observations and source-version vector;
  \item context manifest with required-role coverage, omissions, and conflicts;
  \item typed plan and capability-bound executions;
  \item verifier records for SQL safety, schema, metric, freshness, permissions, and claim support;
  \item structured claims and dependency edges;
  \item report artifact with explicit terminal outcome;
  \item replay package and append-only audit events.
\end{enumerate}
For the main payment request, numeric claims must resolve to the rise from 2.2\% to 7.4\% and Processor~B / Visa concentration; deployment language remains \code{needs\_review}, not causal proof; users without \code{sre} must not receive the restricted incident.

\subsection{Product value gate}

Infrastructure purity is not enough. Before expanding beyond the slice, measure:
\begin{itemize}
  \item reviewer time to locate support for every numeric claim without engineer help;
  \item correction retention into the next applicable run;
  \item zero restricted-item or superseded-item leaks on the seeded suite;
  \item end-to-end run of the payment workflow from CLI and UI with inspectable manifests.
\end{itemize}

\subsection{Customer Journey}

\atxn is the system contract connecting memory, tools, validation, publication, and later invalidation. It is not an ACID transaction across external systems. It creates a truthful \amos commit record after optimistic revalidation of the external versions used.

\subsection{Lifecycle}

\begin{enumerate}
  \item \textbf{BEGIN:} authenticate, bind tenant, classify risk, assign budgets, record policy epoch, create idempotent \atxn ID.
  \item \textbf{OBSERVE:} ask connectors for source versions, freshness, retention class, and access.
  \item \textbf{SELECT:} filter before ranking; reconcile identity, time, authority, supersession; require role completeness.
  \item \textbf{PLAN:} produce a typed plan with declared inputs, tools, budgets, and review obligations.
  \item \textbf{EXECUTE:} issue short-lived capabilities; verify before execution; record parameters, hashes, timings, results.
  \item \textbf{COMPOSE:} construct typed claims from verified result objects, then render the artifact.
  \item \textbf{VERIFY:} validate SQL, schema, metrics, freshness, permissions, claim support, and risk rules.
  \item \textbf{REVALIDATE:} re-read policy epoch and every C1 version; confirm C2 retention.
  \item \textbf{COMMIT:} atomically store artifact metadata, claims, dependencies, verification, replay descriptor, and audit event.
  \item \textbf{DISSEMINATE / REVIEW:} local publication or review task only after evidence commit; external adapters are later.
  \item \textbf{INVALIDATE:} on later source events, traverse reverse dependencies and update validity dimensions independently.
\end{enumerate}

\subsection{Runtime invariants}

\begin{enumerate}
  \item no inaccessible memory enters model-visible context;
  \item every selected object is effective, active, non-superseded, and source-qualified;
  \item every tool call is allowed by both policy and a task-scoped capability;
  \item execution references only declared working-set objects and observed source versions;
  \item numeric claims have computation, data-state, schema, and metric support;
  \item causal or high-impact recommendations remain review-gated;
  \item commit fails or becomes explicitly non-final when governing C1 state changed;
  \item memory writes preserve authority, history, source, and producing transaction;
  \item stale workers cannot commit after a newer fencing token is issued;
  \item evidence commit and external dissemination have separate persistent states;
  \item every final artifact has a declared replay level and complete audit record.
\end{enumerate}

\must Long model and tool work never holds a database transaction open. State transitions use compare-and-swap on \code{state} and \code{state\_seq}. Worker leases carry monotonically increasing fencing tokens. Terminal states are immutable. A state mutation and its outbox event commit together. Specification~A is normative.

\section{Context Compilation}
\label{sec:context}

The context compiler turns an approved, versioned task definition into a permission-safe working set and an auditable manifest.

\must Required roles are stored in task definitions; they are never inferred only by a prompt. The first-slice payment task requires metric definition, active schema, data snapshot, user policy, and review policy.

\subsection{Algorithm}

\begin{enumerate}
  \item derive required and optional roles from the task definition and risk class;
  \item query candidates with tenant and coarse permission pushdown;
  \item remove inaccessible, revoked, expired, and superseded objects \emph{before} ranking;
  \item group by source-qualified logical key and resolve aliases explicitly;
  \item reconcile effective time and authority; surface equal-authority conflicts;
  \item choose at least one valid object for every mandatory role;
  \item fill remaining budget by relevance, authority, freshness, verification state, and diversity;
  \item compact only by producing a non-governing derived object linked to every source;
  \item emit a manifest: selections, omissions, conflicts, warnings, token cost, source versions.
\end{enumerate}

\subsection{Failure and compaction}

Missing mandatory role coverage yields \code{needs\_review} or \code{reject}. Ambiguous approved metrics yield clarification, not a silent pick. Budget pressure never silently drops policy, schema, metric, or data-state roles. Restricted-object counts and identifiers are not revealed to unauthorized users.

\must A compacted summary is non-governing, inherits the strictest source permission, links every source version, and is invalidated when any source changes. It cannot satisfy a required governing role. Specification~C is normative.

\subsection{Ranking for the slice}

The research paper's correctness-first score is sufficient for the first slice:
\[
score(i,q,t,u)=\alpha m(i,q)+\beta a(i)+\gamma f(i,t)+\delta v(i)+\eta s(i)-\lambda c(i,u),
\]
with access violations removed before scoring. Exact coefficient tuning is an engineering artifact; the release gate is zero restricted/superseded selections and stable required-role coverage on frozen fixtures, not leaderboard recall.

\section{Verification, Claims, and Replay}
\label{sec:verify}

\subsection{Concrete verifier for the payment slice}

Verification is structural and semantic, not a proof of truth. Implement the six research-paper checks:

\begin{enumerate}
  \item \textbf{SQL safety:} parseable, read-only root statement; reject writes, DDL, export/copy, and blocked columns such as \code{customer\_email}, \code{payment\_token}, \code{raw\_payload}.
  \item \textbf{Schema validity:} table/column references match schema memory; superseded \code{failure\_reason} fails; current \code{error\_code} passes.
  \item \textbf{Metric compliance:} AST matches approved metric v3---failure numerator aggregate, unfiltered attempt denominator, production filter, test-account exclusion, and \code{event\_time} window. Comments or string literals do not satisfy predicates.
  \item \textbf{Freshness / temporal state:} artifact attaches snapshot or stream state; watermark lag beyond policy emits warning.
  \item \textbf{Permission safety:} inaccessible memory never enters context; capability and policy rechecked at execution and commit.
  \item \textbf{Provenance completeness and review:} numeric claims require query/result, data state, metric, schema, execution hash, and verification; causal/recommendation claims remain \code{needs\_review} unless an approved design applies.
\end{enumerate}

\must The alpha constructs typed claims from verified result objects before rendering prose. Numeric values never originate from model-written text.

\subsection{Claim dependency graph}

For claim $c$:
\begin{equation}
  traceable(c) \not\Rightarrow correct(c) \not\Rightarrow causally\_justified(c).
\end{equation}
The graph proves traceability and change impact. Causal justification remains methodological judgment under review.

\section{Step-by-Step Build Plan}
\label{sec:build}

Advance by exit gates, not calendar optimism. Each phase ends with executable proof in CI.

\subsection{Initial Production Topology}

Deploy the modular monolith as stateless API/runtime replicas behind a load balancer. Use managed PostgreSQL with point-in-time recovery, object storage with versioning, a managed queue, isolated worker pools, secret manager, identity provider, metrics/logs/traces backend, and a private network path to customer sources. Separate development, staging, and production accounts. Infrastructure is declarative and reviewed.

Run interactive and scheduled workloads in separate queues. Partition workers by tool type and risk. Use \atxn IDs and idempotency keys across retries. Database transactions own state transitions; queue acknowledgements occur only after durable state is written. Long operations renew leases and checkpoints.

\subsection{When to Extract Services}

Do not distribute the system prematurely. Extract a component only when one of these conditions is measured: tool workers require a stronger isolation boundary; connector traffic has independent scaling; context indexes exceed primary-database capacity; invalidation fan-out interferes with interactive latency; enterprise deployment requires separate data planes; or team ownership demands an independent release cycle. Preserve the same typed contracts when extracting.

\subsection{Scaling Strategy}

Scale persistent memory and model context independently. Keep structured filters in the database, move large blobs to object storage, and introduce a dedicated lexical/vector candidate index behind the memory interface when needed. Shard by tenant before sharding within a tenant. Maintain reverse dependency indexes for invalidation. Cache only version-addressed, permission-scoped results; policy-epoch changes evict them.

Capacity planning tracks active tenants, memory objects, version rate, dependency edges, scheduled tasks, tool concurrency, result bytes, model tokens, p50/p95/p99 control-path latency, queue age, and invalidation fan-out. Load tests include one-million-object tenants, policy churn, source delay, and noisy neighbors.

\subsection{Concurrency, Allocation, and I/O}

\code{Store} wraps one \code{rusqlite::Connection} in \code{Arc<Mutex<...>>}; cloned services share it and therefore serialize database access within a runtime. WAL permits useful concurrency across separately opened stores. Immediate write transactions, state sequences, and fencing tokens prevent lost updates. All asynchronous-facing database and warehouse work enters an eight-permit \code{spawn\_blocking} lane, so the Tokio executor is not blocked and admission is bounded. Production PostgreSQL adapters still require a measured connection pool, lock/queue telemetry, and the same CAS predicates.

The SQL worker opens a connection per execution, enables \code{query\_only}, disables trusted schema, installs a SQLite progress callback, and materializes only rows that fit the incremental row and byte budgets. Deadline and explicit cancellation both interrupt the statement. Capabilities expire after 60 seconds; constant-time HMAC verification binds issuer, audience, tenant, \atxn, plan, step, subject, tool, source, operations, relations, limits, policy epoch, fence, and token ID at both connector and worker boundaries. Production remote workers must additionally obtain short-lived source credentials without ambient process access.

Hashes are SHA-256 over fallibly serialized values; serialization failures propagate and cannot alias empty content. Runtime entity IDs are typed UUIDv7 values, while durable checkpoint and event identities use typed prefixes plus 128 bits of the serialized-value digest. JSON payloads keep the local adapter simple; indexed scalar columns remain authoritative for concurrency, filtering, and lifecycle state.

\subsection{Performance Gates}

Optimization is accepted only with a workload, correctness oracle, and before/after evidence. The paid-pilot gate records p50/p95/p99 for admission, context compilation, verification, execution, evidence commit, review commit, job acquisition, invalidation page, and replay; peak resident memory; allocations per selected context item and output row; database lock/pool wait; rows and bytes scanned/returned; and outbox/job age. Benchmarks cover $10^4$, $10^5$, and $10^6$ memory objects, skewed permission labels, conflicting authority versions, large SQL outputs, 100--250-claim invalidation pages, expired lease storms, duplicate admissions/reviews, and concurrent tenants. No optimization may weaken permission-before-ranking, source-version immutability, compare-and-swap, fencing, atomic evidence, or claim-level provenance.

\section{Architecture Decisions and Non-Goals}

Key decisions are explicit:
\begin{itemize}
  \item typed memory, permission-first context, A-TXN state machine with CAS and fencing;
  \item read-only SQL worker with signed capabilities; statistics and chart workers;
  \item AST verification for safety, schema, metric predicates, freshness, and claim support;
  \item structured claims, dependency edges, review corrections, invalidation, and level-3 replay;
  \item Analysis Workspace, Memory Studio, Review Queue, and Operations Console without a JavaScript runtime;
  \item scenario fixtures for payment failure, with subscription churn and warehouse quality available for Phase~6.
\end{itemize}

Production PostgreSQL/RLS, object storage, KMS/secret management, SSO, and customer-specific warehouse connectors remain deployment adapters. They are not missing domain functions and must not rewrite the contracts in Appendices A--F.

\section{Architecture Decisions}

\begin{itemize}
  \item modular monolith first, with internal service contracts and one authoritative transaction store;
  \item keep raw data in source systems; move bounded results or retained snapshots only under policy;
  \item enforce permissions before retrieval ranking and again at execution and commit;
  \item treat models as untrusted, replaceable processors with no ambient credentials;
  \item use optimistic cross-source validation rather than nonexistent global ACID;
  \item construct typed claims before prose; attach provenance at claim granularity;
  \item separate evidence commit from external dissemination;
  \item gate causal and high-impact conclusions on human or approved methodological review;
  \item add autonomy per task class, never as a platform-wide switch;
  \item cut enterprise surface area until the payment slice is boringly reliable and valuable.
\end{itemize}

\section{Definition of Done}
\label{sec:done}

Before claiming the first production slice is ready for a design partner, answer yes to every item:

\begin{enumerate}
  \item Can an operator reconstruct why every selected memory object entered context and why every omitted mandatory role caused a visible outcome?
  \item Can permission revocation between retrieval and publication prevent commit without relying on the model?
  \item Can every numeric statement in a final payment artifact resolve to query, result hash, data version, metric, schema, and verification?
  \item Can a reviewer correction affect the next applicable task while the original artifact remains immutable?
  \item Can a metric or schema change identify and reclassify every dependent claim?
  \item Can duplicate requests or retries occur without duplicate local publication?
  \item Can a worker be killed without exposing long-lived credentials or corrupting transaction history?
  \item Do the seeded security probes for prompt injection, memory poisoning, restricted retrieval, and blocked SQL all fail closed?
  \item Can a second engineer run \code{cargo run -- seed}, \code{cargo run -- serve}, and the payment CLI path without tribal knowledge?
  \item Does the workflow save measurable analyst/reviewer time on the payment review compared with the team's prior process?
\end{enumerate}

A no answer is an architectural or product gap, not documentation debt. The corresponding phase remains incomplete until the gap has an owner, contract, test, and owner-visible status.

\section{Conclusion}

\amos becomes a product when persistent company knowledge, model reasoning, tool execution, verification, human judgment, and artifacts are joined by one enforceable operating contract. Build that contract for one governed read-only workflow first. Make every outcome and dependency observable. Add durable review, invalidation, and replay. Only then earn connectors, tenancy hardening, platform APIs, and autonomy.

If the team follows the first-slice scope, component ownership, build gates, and Appendices A--F---and keeps repository schemas as the executable truth---it can build \amos end to end without leaving the hardest correctness decisions implicit or drowning in premature enterprise surface area.

\subsection{Compare-and-swap and fencing}

\must Each database interaction is a short transaction. Transition code performs compare-and-swap on tenant, \atxn ID, current state, and \code{state\_seq}; success increments the sequence. A zero-row update is a concurrency conflict.

\begin{lstlisting}[style=contract]
UPDATE atxn_transactions
SET state = :next_state,
    state_seq = state_seq + 1,
    updated_at = clock_timestamp()
WHERE tenant_id = :tenant_id
  AND atxn_id = :atxn_id
  AND state = :expected_state
  AND state_seq = :expected_seq
  AND terminal = false;
\end{lstlisting}

Job acquisition uses row locking with skip-locked semantics, increments \code{fencing\_token}, and commits before work begins. Every checkpoint, execution, object promotion, outbox completion, and publication attempt compares that fence. Terminal records reject updates at repository and database layers.

\must Every mutation and its outbox event share one database commit. Side effects use a stable
tenant-scoped idempotency key for each step effect; publication uses
\code{tenant/publication\_id}. Duplicate requests return the original resource when the request hash
matches and \code{IDEMPOTENCY\_CONFLICT} when it differs.
